# BOSTONCOLLECT

Полноценная музыкальная веб-платформа на базе `Next.js + NestJS + PostgreSQL`, в которой пользователь может:

- регистрироваться и входить в систему
- искать треки и альбомы
- открывать страницу альбома
- сохранять треки и альбомы в личную коллекцию
- писать сообщения в общий чат
- работать с админ-панелью при наличии роли `admin`

Проект разделён на две части:

- `frontend/` — клиентская часть на `Next.js`
- `backend/` — серверная часть на `NestJS`

## Идея проекта

Изначально проект строился как музыкальная платформа с базовой авторизацией, поиском и выводом данных из API. Дальше архитектура была расширена до полноценного приложения с ролями, доступами, личной коллекцией, общим чатом и административным управлением каталогом.

Конечная цель разработки была не просто "сделать страницы", а собрать целостную систему, где:

- фронтенд и бэкенд разделены по ответственности
- данные берутся из PostgreSQL, а не хранятся в моках
- авторизация и роли проходят через серверную проверку
- критические маршруты закрыты как на клиенте, так и на сервере
- администратор может управлять музыкальным каталогом безопасно и централизованно

## Архитектура проекта

### Frontend

На клиенте реализованы:

- авторизация и хранение токена в cookie
- глобальный auth-state через `Zustand`
- отдельный store для личной коллекции
- защита маршрутов через `frontend/proxy.ts`
- страницы каталога, поиска, коллекции, чата и админки

Основные файлы фронтенда:

- `frontend/app/` — страницы приложения
- `frontend/components/` — переиспользуемые UI-компоненты
- `frontend/store/` — глобальные состояния авторизации и коллекции
- `frontend/lib/api.ts` — единая обёртка над API-запросами
- `frontend/proxy.ts` — защита маршрутов

### Backend

На сервере логика разделена по модулям:

- `auth` — регистрация, логин, JWT, получение профиля
- `albums` — получение альбомов и страницы конкретного альбома
- `search` — поиск по каталогу
- `collections` — личная коллекция пользователя
- `chat` — общий чат
- `admin` — админские CRUD-операции

Сервер работает поверх PostgreSQL через Prisma и использует реальные таблицы из базы, а не отдельную искусственную схему приложения.

## Работа с ролями

Роли в проекте берутся из PostgreSQL, а именно из связки:

- `app_roles` — справочник ролей
- `app_user_roles` — связка пользователя и роли

Текущая модель ролей:

- `admin` — id `1`
- `user` — id `2`
- `guest` — id `3`

Важно: система не хардкодит роль пользователя в интерфейсе. Роль определяется на сервере.

### Как это работает

1. При логине сервер читает пользователя из `app_users`.
2. Вместе с пользователем подтягиваются его роли из `app_user_roles -> app_roles`.
3. Эти роли записываются в JWT payload.
4. На следующих запросах сервер проверяет JWT и использует роли из токена для доступа к защищённым маршрутам.
5. Для админки дополнительно используется `RolesGuard`.

Таким образом, источник правды по ролям — база данных, а JWT хранит подписанный снимок роли после входа пользователя.

Если роль поменять прямо в БД, новая роль начнёт действовать после следующего логина пользователя.

### Пользовательские страницы

- `/` — главная страница с каталогом альбомов
- `/search` — поиск по трекам и альбомам
- `/album/[id]` — страница конкретного альбома
- `/collection` — личная коллекция пользователя
- `/chat` — общий чат
- `/login` — вход

- `/library` — редирект на `/collection` для совместимости

### Административная страница

- `/admin` — доступна только пользователю с ролью `admin`

Раздел `artists` был убран как отдельная пользовательская страница, потому что по бизнес-логике он не нужен.

## Возможности администратора

Администратор может:

- добавлять артистов
- редактировать артистов
- удалять артистов
- добавлять треки
- редактировать треки
- удалять треки
- создавать альбомы
- редактировать альбомы
- задавать порядок треков в альбоме
- назначать артистов трекам внутри альбома
- удалять альбомы
- удалять сообщения в общем чате

Это реализовано через отдельный модуль `admin` на сервере и защищено ролью `admin`.

## Личная коллекция

У каждого пользователя есть отдельная коллекция.

Коллекция хранится в таблицах:

- `collections`
- `collection_albums`
- `collection_songs`

Пользователь может из интерфейса:

- сохранить альбом
- убрать альбом из коллекции
- сохранить трек
- убрать трек из коллекции

Кнопки добавления в коллекцию доступны на:

- главной странице
- странице поиска
- странице альбома
- странице коллекции

При регистрации пользователю автоматически создаётся его личная коллекция.

## Общий чат

В проекте реализован общий чат для всех авторизованных пользователей.

Возможности:

- любой авторизованный пользователь может отправить сообщение
- пользователь может удалить собственное сообщение
- администратор может удалить любое сообщение

Сообщения хранятся в таблице `chat_messages`.

## Безопасность проекта

Безопасность в проекте строится на нескольких уровнях.

### 1. Пароли не хранятся в открытом виде

При регистрации пароль хэшируется через `bcrypt` и только потом записывается в базу.

### 2. JWT-подпись

После логина пользователь получает подписанный JWT. На защищённых маршрутах сервер валидирует подпись токена.

### 3. Защита маршрутов на сервере

Для защищённых endpoint используются:

- `JwtAuthGuard` — проверка, что пользователь авторизован
- `RolesGuard` — проверка, что у пользователя есть нужная роль

Это главная защита. Даже если кто-то вручную откроет страницу на фронтенде, без нужных прав сервер не даст доступ к admin-endpoint.

### 4. Защита маршрутов на фронтенде

В `frontend/proxy.ts` закрыты:

- главная
- поиск
- альбомы
- коллекция
- чат
- админка

Если нет токена, пользователь перенаправляется на `/login`.

Для `/admin` дополнительно проверяется наличие роли `admin` в токене.

### 5. DTO и валидация

На сервере используется `ValidationPipe` и DTO-классы.

Это даёт:

- фильтрацию лишних полей
- валидацию входящих данных
- защиту от несанкционированной передачи произвольных полей

Например, клиент не может просто так прислать дополнительное поле и ожидать, что сервер его примет.

### 6. Роли не доверяются клиенту

Роль пользователя не задаётся фронтендом. Она читается сервером из базы и только потом попадает в подписанный JWT.

### 7. Ограничение прав по операциям

Даже внутри чата и админских действий действуют дополнительные ограничения:

- обычный пользователь не может удалить чужое сообщение
- неадмин не может вызывать admin-endpoint
- коллекция работает только с текущим авторизованным пользователем

## Как разрабатывался проект

Разработка шла поэтапно:

1. Была собрана база: авторизация, логин, поиск и вывод данных с API.
2. Затем была переработана auth-логика, чтобы роли брались из PostgreSQL через реальные таблицы `app_roles` и `app_user_roles`.
3. После этого был добавлен защищённый серверный слой для админских действий.
4. Затем появились недостающие пользовательские разделы: коллекция и общий чат.
5. После функционального завершения был обновлён дизайн: зелёная стилистика заменена на фиолетовую, а бренд проекта изменён на `BOSTONCOLLECT`.
6. В конце проект был проверен через сборку фронтенда и бэкенда, а также через линтер и TypeScript-проверку.

Подход в разработке был такой:

- сначала сделать корректную серверную модель доступа
- потом подключить к ней фронтенд
- затем улучшить пользовательский опыт и визуальную часть
